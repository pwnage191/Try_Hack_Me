### Observing the Network Flow
- I visited the web application and found an MD2PDF converter with a text input field.
- To understand how the application handled user input, I submitted a simple HTML payload: `<b>Hello all</b>`.
- Since Markdown renderers often support embedded HTML, I wanted to verify whether HTML content would be rendered in the generated PDF.
- After clicking the **Convert to PDF** button, a new tab opened and displayed the generated PDF containing the bold text.
- I then inspected the application using the browser's Developer Tools.
- The application behaved like a modern web application, where the initial request loaded the required JavaScript and CSS resources.
- When the **Convert** button was clicked, the client-side JavaScript automatically sent a `POST` request to the `/convert` endpoint.
- I also noticed the `Referrer-Policy: strict-origin-when-cross-origin` header. This policy controls how referrer information is shared between
  origins, but it does not restrict requests to internal resources.

### Discovering the PDF Conversion Tool
- To better understand the backend behavior, I reproduced the request using `curl` and manually supplied the same headers used by the browser.
- The response matched the behavior observed through the web interface.
- While inspecting the generated PDF metadata, I discovered that the application uses `wkhtmltopdf` to convert HTML content into PDF documents.
- I also observed that the MD2PDF service was accessible on both port `80` and port `5000`, exposing the same `/convert` functionality.
- Based on these observations, I concluded that the workflow was:
  - The user submits content to the `/convert` endpoint.
  - The server passes the content to `wkhtmltopdf`.
  - The `/convert` endpoint launches a `wkhtmltopdf` subprocess, which renders the supplied HTML, generates a PDF, saves it to a temporary location,
    terminates, and then allows the main application to read the generated PDF and return it to the user.

### Accessing the Admin Panel
- Since user-supplied HTML is rendered by `wkhtmltopdf`, I investigated whether the renderer could access internal resources.
- When the server processes our request, `wkhtmltopdf` runs as a subprocess and renders the supplied HTML.
- Because `wkhtmltopdf` is capable of making network requests while rendering content, it can be abused to access internal services that are not
  directly reachable by external users.
- This behavior effectively creates a Server-Side Request Forgery (SSRF) primitive.
- To leverage this, I embedded the following HTML payload:
  ```html
  <iframe src="http://localhost:5000/admin"></iframe>
  ```
- When wkhtmltopdf rendered the HTML, it processed the iframe and issued a request to http://localhost:5000/admin from the server itself.
- Since the request originated from localhost, the access restriction was bypassed.
- The response from the internal /admin endpoint was rendered inside the generated PDF.
- Finally, the PDF was returned to the browser, allowing me to view the contents of the otherwise inaccessible admin panel.
